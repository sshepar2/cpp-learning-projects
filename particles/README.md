# Particles — C++ OOP Practice Project

A small C++ project demonstrating object-oriented design, static code analysis, and CMake build system setup. Built as a practice exercise covering the core C++ concepts for system critical design. 

## Project Structure

```
particles/
├── CMakeLists.txt          # top level build configuration
├── src/
│   ├── CMakeLists.txt      # defines library and executable
│   ├── particle.cpp        # class implementations
│   ├── simulation.cpp      # printSimulationSummary implementation
│   └── main.cpp            # entry point
├── include/
    ├── particle.h          # class declarations + include guards
    ├── simulation.h        # printSimulationSummary declaration
├── bin/                    # compiled executable (generated)
└── build/                  # CMake build artifacts (generated)
```

## What It Demonstrates

### Object-Oriented Design
- Abstract base class `Particle` with pure virtual `describe()` — prevents direct instantiation and enforces that all derived classes implement the method
- Virtual destructor on base class to ensure correct cleanup when deleting derived objects through a base pointer
- Two derived classes: `ClassicalParticle` and `QuantumParticle`, each overriding `describe()` and where applicable `kineticEnergy()`
- Polymorphism via `std::vector<Particle*>` — `printSimulationSummary` operates on any particle type without knowing the specific derived type at compile time
- Member initialization via constructor initialization lists rather than assignment in the constructor body
- `std::array<float, 3>` for fixed-size position and velocity data rather than `std::vector`, appropriate for performance-sensitive code

### Project Structure
- Separation of class declarations (`.h`) from implementations (`.cpp`)
- Include guards (`#ifndef / #define / #endif`) in all header files to prevent duplicate declarations
- `simulation.cpp` kept separate from particle classes — free functions decoupled from class definitions

### Build System
- CMake out-of-source build (`-Bbuild`) keeps generated files isolated from source
- `add_library` compiles particle and simulation sources into an intermediate static library
- `add_executable` links `main.cpp` against the library
- `PRIVATE` vs `PUBLIC` visibility on `target_include_directories` and `target_compile_options` controls whether properties propagate to linking targets
- `CMAKE_RUNTIME_OUTPUT_DIRECTORY` set to `bin/` to separate the final binary from build artifacts
- `CMAKE_EXPORT_COMPILE_COMMANDS=ON` generates `compile_commands.json` for use with static analysis tools

### Static Code Analysis
- Cppcheck run via `compile_commands.json` for accurate analysis with correct include paths and compiler flags
- `--suppress=missingIncludeSystem` suppresses noise from standard library headers while preserving warnings for project headers
- Clean output — no warnings or errors on final version of code

## Building

#### Option 1: Using `cmake` directly

```bash
cmake -S. -Bbuild -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
cmake --build build -j
```

#### Option 2: Using bash script (just the runs `cmake` comamnds in option 1)

The build commands can also be run using the provided shell script,
```bash
./build.sh
```

#### Option 3: Using `make`

There is a Makefile in which `build` and `clean` are defined. The executable can be generated by running,
```bash
make
```

Executable is written to `bin/simulation`.

## Running Cppcheck

```bash
cppcheck --project=build/compile_commands.json --suppress=missingIncludeSystem
```

This command can also be run via 
```bash
./check.sh
```

## Cleaning

```bash
rm -rf build/ bin/
```

or

```bash
make clean
```

## Key C++ Concepts Referenced

- Pure virtual functions and abstract base classes
- Virtual destructors
- Constructor initialization lists
- `std::array` vs `std::vector` for fixed-size data
- `override` keyword
- `const` correctness on getters
- `std::unique_ptr` and heap vs stack allocation
- Static vs shared libraries (`.a` vs `.so`)
- Include guards and translation units
- Compiler warning flags (`-Wall`, `-Wextra`)
- `gcc` / `g++` compilation pipeline: preprocessing, compilation, linking
